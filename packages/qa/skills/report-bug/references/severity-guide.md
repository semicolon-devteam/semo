# QA Severity Guide

> QA 테스트용 버그 심각도 기준 가이드

## 심각도 레벨

| 레벨 | 라벨 | 설명 | QA 대응 | 개발 대응 시간 |
|------|------|------|---------|---------------|
| **Critical** | `critical` | 서비스 전체 불가, 데이터 손실 위험 | 즉시 보고 | 즉시 |
| **High** | `high` | 주요 기능 장애, 대안 없음 | 테스트 중단 | 24시간 내 |
| **Medium** | `medium` | 기능 일부 문제, 대안 있음 | 테스트 계속 | 1주일 내 |
| **Low** | `low` | 사소한 이슈, UX 개선 수준 | 테스트 계속 | 스프린트 내 |

## QA 관점 상세 기준

### Critical (치명적) - 즉시 보고

**판단 기준**:
- 테스트 진행 자체가 불가능
- 데이터 무결성 문제
- 보안 취약점 발견
- 결제/금전 관련 오류

**QA 대응**:
1. 즉시 테스트 중단
2. 개발팀에 긴급 보고
3. 버그 리포트 생성
4. 재현 가능한 증거 확보

**예시**:
- API 서버 500 에러로 모든 요청 실패
- 결제 금액 계산 오류
- 다른 사용자 데이터 노출

### High (높음) - 테스트 중단 고려

**판단 기준**:
- 핵심 플로우 테스트 불가
- 해당 기능 전체가 작동 안 함
- 다수의 테스트 케이스 블로킹

**QA 대응**:
1. 해당 기능 테스트 중단
2. 다른 기능으로 테스트 전환
3. 버그 리포트 생성 후 추적

**예시**:
- 로그인 기능 전체 실패
- 게시글 목록 로딩 안 됨
- 검색 기능 응답 없음

### Medium (중간) - 테스트 계속

**판단 기준**:
- 일부 기능에 문제
- 우회 방법으로 테스트 가능
- 특정 조건에서만 발생

**QA 대응**:
1. 버그 기록 후 테스트 계속
2. 우회 방법으로 나머지 테스트 진행
3. 테스트 완료 후 버그 리포트 정리

**예시**:
- 특정 브라우저에서만 UI 깨짐
- 긴 텍스트 입력 시 레이아웃 문제
- 간헐적 타임아웃 (재시도 시 성공)

### Low (낮음) - 기록 후 진행

**판단 기준**:
- 기능에 영향 없음
- UI/UX 개선 사항
- 오타, 문구 오류

**QA 대응**:
1. 메모로 기록
2. 테스트 완료 후 일괄 리포팅
3. 우선순위 낮게 분류

**예시**:
- 버튼 텍스트 오타
- 색상 가이드 불일치
- 불필요한 공백

## 재현율과 심각도

| 재현율 | 심각도 영향 |
|--------|-----------|
| 100% (항상 재현) | 심각도 유지 또는 상향 |
| 50-99% (자주 재현) | 심각도 유지 |
| 10-49% (가끔 재현) | 심각도 하향 고려 |
| < 10% (드물게 재현) | 심각도 하향, 모니터링 권장 |

## 자동 라벨링

```bash
# Critical
--label "bug,qa-found,critical,priority:urgent"

# High
--label "bug,qa-found,high,priority:high"

# Medium
--label "bug,qa-found,medium"

# Low
--label "bug,qa-found,low"
```

## 리포팅 우선순위

QA 테스트 완료 후 버그 리포팅 순서:

1. **Critical** → 즉시 (테스트 중에도)
2. **High** → 해당 기능 테스트 완료 직후
3. **Medium** → 일일 테스트 완료 후
4. **Low** → 이터레이션 종료 시 일괄
