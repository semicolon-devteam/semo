# MVP Mode

> 빠른 검증을 위한 패스트트랙 개발 모드

## Overview

MVP 모드는 **속도 우선**으로 PoC, 프로토타입, 빠른 검증을 위한 개발에 적합합니다.

## 적용되는 간소화

### 테스트
- [ ] 테스트 작성 **선택적** (필수 아님)
- [ ] 코드 커버리지 요구 없음
- [ ] E2E 테스트 생략 가능

### 코드 품질
- [ ] 코드 리뷰 **생략 가능**
- [ ] DDD 4계층 → **2계층 허용** (hooks + components)
- [ ] 타입 정의 간소화 (`any` 일부 허용)
- [ ] 주석/문서화 최소화

### 아키텍처
- [ ] 복잡한 추상화 지양
- [ ] 단일 파일에 여러 컴포넌트 허용
- [ ] 유틸리티 함수 인라인 허용

### CI/CD
- [ ] 린트 경고 허용 (에러만 차단)
- [ ] 빌드 통과만 확인
- [ ] 스테이징 배포 생략 가능

## 유지되는 필수 규칙

### 절대 생략 불가
- [x] **기본 폴더 구조** (src/, components/, hooks/ 등)
- [x] **네이밍 컨벤션** (camelCase, PascalCase)
- [x] **Git 커밋 규칙** (feat:, fix:, docs: 등)
- [x] **보안 규칙** (비밀번호 하드코딩 금지, .env 사용)
- [x] **ESLint 에러** (경고는 허용, 에러는 수정)

### 최소한의 품질
- [x] TypeScript 컴파일 통과
- [x] 기본 빌드 성공
- [x] 크리티컬 버그 없음

## 사용 방법

### Orchestrator에 모드 전달
```markdown
[eng/nextjs --mode=mvp] 로그인 페이지 빠르게 만들어줘
```

### 명시적 선언
```markdown
MVP 모드로 개발해줘. 테스트는 나중에 추가할게.
```

## MVP → Production 마이그레이션 체크리스트

MVP 개발 완료 후 Production으로 전환 시:

### 구조 정리
- [ ] 2계층 → 4계층 리팩토링
- [ ] 타입 정의 강화 (`any` 제거)
- [ ] 컴포넌트 분리

### 품질 강화
- [ ] 테스트 추가 (커버리지 80%)
- [ ] 에러 핸들링 강화
- [ ] 로깅 추가

### 문서화
- [ ] API 문서 작성
- [ ] 아키텍처 문서 작성
- [ ] 운영 가이드 작성

## 적합한 상황

| 상황 | MVP 모드 적합성 |
|------|----------------|
| PoC 검증 | O 매우 적합 |
| 해커톤 | O 매우 적합 |
| 빠른 피드백 수집 | O 적합 |
| 실서비스 개발 | X 부적합 |
| 장기 유지보수 | X 부적합 |
