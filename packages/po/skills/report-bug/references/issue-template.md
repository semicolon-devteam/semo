# Bug Report Issue Template

> 버그 리포트 이슈 템플릿 상세 가이드

## 전체 템플릿

```markdown
## 🐛 버그 리포트

### 요약
{버그에 대한 간단한 설명 - 1~2문장}

### 재현 단계
1. {환경/페이지로 이동}
2. {특정 동작 수행}
3. {추가 동작}
4. {버그 발생 확인}

### 기대 동작
{정상적으로 동작해야 하는 방식}

### 실제 동작
{현재 발생하는 문제}

### 심각도
- [ ] Critical - 서비스 불가
- [ ] High - 주요 기능 장애
- [ ] Medium - 기능 일부 문제
- [ ] Low - 사소한 이슈

### 환경
- 발생 위치: {페이지 URL 또는 API 경로}
- 관련 기능: {기능명}
- 브라우저/디바이스: {해당 시}
- 스크린샷: {있다면 첨부}

### 추가 정보
{로그, 에러 메시지, 관련 이슈 등}

## ✅ Acceptance Criteria
- [ ] {완료 조건 1 - 버그 수정 후 기대 동작}
- [ ] {완료 조건 2 - 추가 확인 사항}

## 🧪 QA 테스트 요구사항
- [ ] {사용자 시나리오 테스트 1 - 재현 단계 검증}
- [ ] {사용자 시나리오 테스트 2 - 엣지 케이스}
- [ ] {회귀 테스트 - 관련 기능 정상 동작}

## 🌿 Branch

`fix/{issue-number}-{slug}`

---
🤖 SEMO report-bug Skill로 자동 생성됨
```

## 필드별 가이드

### 요약

- 버그의 핵심을 1~2문장으로 요약
- "~가 ~할 때 ~가 발생함" 형식 권장

**예시**:
- "로그인 버튼 클릭 시 무한 로딩 발생"
- "게시글 작성 후 목록에 반영되지 않음"

### 재현 단계

- 버그를 재현할 수 있는 구체적인 단계
- 누구나 따라할 수 있도록 명확하게 작성
- 4~6단계 권장

**예시**:
```markdown
1. 로그인 페이지(/login)로 이동
2. 이메일: test@example.com, 비밀번호: test1234 입력
3. 로그인 버튼 클릭
4. 무한 로딩 스피너 확인 (10초 이상 지속)
```

### 기대 동작 vs 실제 동작

| 구분 | 설명 |
|------|------|
| 기대 동작 | 정상적으로 동작해야 하는 방식 |
| 실제 동작 | 현재 발생하는 문제 상황 |

**예시**:
- 기대: "로그인 성공 후 메인 페이지로 이동"
- 실제: "로딩 스피너가 무한히 표시되며 페이지 이동 없음"

## Acceptance Criteria 가이드

- 버그 수정 완료의 명확한 기준 정의
- 체크리스트 형태로 검증 가능하게 작성
- 기대 동작 섹션과 연계

**예시**:
```markdown
## ✅ Acceptance Criteria
- [ ] 로그인 버튼 클릭 시 3초 이내 응답
- [ ] 로그인 성공 후 메인 페이지로 리다이렉트
- [ ] 잘못된 비밀번호 입력 시 에러 메시지 표시
```

## QA 테스트 요구사항 가이드

- 사용자 관점의 시나리오 테스트
- 재현 단계 기반의 검증
- 엣지 케이스 및 회귀 테스트 포함

**예시**:
```markdown
## 🧪 QA 테스트 요구사항
- [ ] 재현 단계대로 정상 로그인 가능 확인
- [ ] 빈 이메일/비밀번호로 제출 시 validation 확인
- [ ] 소셜 로그인 기능 영향 없음 확인 (회귀)
```

## 브랜치 네이밍 규칙

```
fix/{issue-number}-{slug}
```

| 요소 | 설명 | 예시 |
|------|------|------|
| `fix/` | 버그 수정 접두사 | 고정 |
| `{issue-number}` | 이슈 번호 | `42` |
| `{slug}` | 버그 요약 slug | `login-infinite-loading` |

**예시**:
- `fix/42-login-infinite-loading`
- `fix/108-post-not-showing`
- `fix/215-payment-error`
